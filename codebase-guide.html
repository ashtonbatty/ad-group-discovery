Vendor AD Group Discovery, explained from input files to reports.

This project finds Active Directory groups that appear to belong to, or be used by, a vendor. It reads small CSV files, queries one or more domains, scores the evidence it finds, and writes reports that show each group, its members, and the reasons it was included.

The short version

What this project does

Vendor access in Active Directory is rarely marked by one perfect field. A vendor group might contain vendor users, be managed by one of those users, have a vendor name in its group name, live in a vendor-named OU, or mention a vendor account in its description. This codebase combines those clues and turns them into a confidence-ranked list.

The project is packaged as a PowerShell module named VendorAdGroupDiscovery. The exported function is Find-VendorAdGroup, and the root script Find-VendorAdGroup.ps1 is a convenience wrapper that imports the module and forwards the parameters. In normal use, you run the wrapper from a machine that can load the RSAT ActiveDirectory module and read the target domains. 

./Find-VendorAdGroup.ps1 `
  -UsersCsv samples/users.csv `
  -DomainsCsv samples/domains.csv `
  -KeywordsCsv samples/keywords.csv `
  -KnownGroupsCsv samples/known.csv `
  -ExcludeGroupsCsv samples/exclude.csv `
  -OutputDirectory ./out

The command returns an object with Results and Summary, but its main purpose is to write files into the output directory. By default it writes CSV reports, an HTML report, and a console summary. You can narrow the output with -Formats, restrict candidates to security groups with -SecurityGroupsOnly, and raise the report threshold with -MinimumConfidence.

Execution flow

How one discovery run moves through the code

1. Read CSV input Read-DiscoveryInput validates required columns and returns one object containing users, domains, keywords, known groups, and excluded groups.
2. Query AD Get-AdDiscoveryData resolves vendor users, searches for likely groups, hydrates candidate groups, and builds a DN index.
3. Score evidence Invoke-DiscoveryEngine calls the matching functions, expands nested relationships, and filters by confidence.
4. Shape display data Still inside the engine, Resolve-ResultDisplay turns DNs into readable names and marks vendor users in member lists, then results are sorted.
5. Write reports The report layer writes group CSV, member CSV, user reports, HTML, and a console summary.

The top-level function in src/Find-VendorAdGroup.ps1 is intentionally easy to read. It does not contain the matching rules itself. Instead, it wires together private functions: read input, create the output folder, query Active Directory, run the engine, build a summary, and write the requested report formats. That makes it a good starting point when you want to understand the project without getting lost in implementation detail.

How the repository is organized

The module loader, VendorAdGroupDiscovery.psm1, dot-sources every .ps1 file under src/ and exports only Find-VendorAdGroup. Dot-sourcing means each function is loaded into the module scope, so private helpers can call each other even though only one function is public.

Why the layers exist

Four conventions hold the design together

The folder map shows what the layers are. These four rules explain why they are split that way. Each one exists to solve a concrete problem, and keeping them in mind makes the rest of the codebase feel intentional rather than arbitrary.

A thin Active Directory boundary Only Get-AdDiscoveryData is allowed to call AD cmdlets. Everything it learns is packed into one plain data object. That single choke point is what lets the whole engine run on Linux pwsh with no directory at all.
A pure engine and report layer Because the AD layer hands downstream code plain objects, every matching and report function is a pure transformation of data in, data out. Purity is precisely what makes the fixture and mock testing story below possible.
Output is treated as untrusted Group names, descriptions, and info can be edited by people the script never controls. Protect-CsvCell defuses spreadsheet formula injection and Write-HtmlReport HTML-encodes every value, so a hostile group name cannot become a payload.
Shared rules live in one function Where fetching and scoring must agree — for example which groups may seed trusted-name searches — the decision lives in a single helper such as Test-TrustedNameSource that both sides call, rather than being duplicated and drifting apart.

One idiom worth knowing before you read the engine

Several engine functions return their result array with a leading comma, as in ,$results.ToArray(). This stops PowerShell from unrolling a single-element or empty array on the way out. Call these functions with a plain assignment; wrapping the call in an extra @( ) nests the array inside another array and is a common source of confusion.

Configuration

The five CSV files are the project control panel

The code does not hard-code a vendor name or domain list. Instead, each run is configured by CSV. Read-DiscoveryInput checks that each file exists and that required columns are present. It also handles the empty-file case carefully by reading the header line directly, so a file with headers but no rows can still pass column validation.

users.csv Requires SamAccountName. Optional UUserId and DisplayName add more context. AD lookups later add mail, SID, DN, and account status fields.
domains.csv Requires Domain. Optional Server lets you choose a specific domain controller; optional Name is a friendly label.
keywords.csv Contains vendor words or phrases. These are used against group names, descriptions, info fields, and OU names.
known.csv Requires Domain and Identity. Known groups are treated as confirmed evidence and can seed related searches.
exclude.csv Requires Domain and Identity. Matching groups are skipped even if they otherwise look relevant.

Identity in known.csv and exclude.csv may be a group name or a distinguished name. Internally, the project normalizes these through Get-GroupLookupKey, so comparisons are consistent by domain.

Directory access

The AD layer tries to avoid expensive, unfocused searches

Get-AdDiscoveryData is the largest function because it owns the costly part of the job: talking to Active Directory. It first resolves the vendor users from users.csv in each domain, batching sAMAccountName LDAP filters so the query strings stay at a reasonable size. It skips suspicious account names containing characters that would make filter construction risky.

Once the vendor users are known, the function searches for candidate groups in several ways: direct memberships from each vendor user's memberOf, groups whose member contains a vendor user or foreign security principal SID, keyword hits in group metadata, groups under keyword-matching OUs, description hits for vendor account tokens, explicitly known groups, nested parent groups, and related groups mentioned by trusted group names.

The important idea is that the AD layer gathers a focused candidate set. It does not simply download every group in every domain and hope the later engine can sort it out. It also caches users, groups, and member objects so report shaping does not repeatedly fetch the same DN.

Why the trusted-name loop exists

Some groups are connected indirectly. For example, a confirmed vendor group might be named Acme Admins, and another group might mention that name in its description. The code only lets sufficiently trusted groups seed those description-name searches. That rule is shared by Test-TrustedNameSource, Get-AdDiscoveryData, and Expand-VendorGroupClosure, so fetching and scoring agree with each other.

Matching engine

Evidence becomes reasons, reasons become confidence

The engine works on hydrated group objects, not raw AD cmdlet output. Find-CandidateGroups loops through each candidate group, skips excluded groups, checks whether the group is known, and collects match reasons from small helper functions. Each reason has a Pattern such as NameKeyword, Owner, or MemberVendorUser, plus a value explaining what matched.

Name and OU signals Get-NameMatchReason and Get-ContainerMatchReason look for configured keywords in the group name and OU components of the DN.
Description signals Get-DescriptionMatchReasons checks description and info for keywords and vendor user tokens such as SAM, UUserId, or mail.
Owner signals Get-OwnerMatchReason compares managedBy against a vendor-user index keyed by DN and SID.
Member signals Get-MemberMatchReasons marks direct vendor users in a group's member list. These are weaker because shared groups often contain vendor users too.

Get-MatchConfidence is the scoring table. Known groups are always Confirmed. Otherwise, strong reasons are worth 3 points, medium reasons are worth 2 points, and each direct vendor-user member is worth 1 point up to a maximum of 3. That cap is intentional: a group with many vendor users may still be a shared group, so membership alone should not become unbounded proof.

ConfirmedListed in known.csv.
HighScore 3 or higher, such as a keyword in the name, OU, owner, or several members.
MediumScore 2, such as description evidence or a nested vendor group.
LowScore 1, usually a single direct vendor-user member.

After initial scoring, Expand-VendorGroupClosure runs up to 25 rounds. It promotes relationships that can only be seen after one group is already trusted: parent groups that contain confirmed or medium-confidence vendor groups receive a NestedVendorGroup reason, and descriptions that mention trusted group names receive a DescriptionGroup reason. Each added reason recalculates the score and confidence.

Output

The report layer is separate from discovery logic

Once the engine selects and sorts results, the report functions focus on presentation. They do not decide what is a vendor group. That separation keeps the matching behavior testable without needing to inspect generated HTML or CSV.

vendor-group-discovery.csv One row per selected group. Includes domain, name, confidence, score, source, owner, member counts, parent groups, AD metadata, DN, and match reasons.
vendor-group-discovery-members.csv One row per direct member of each reported group. Members are classified as known vendor users, nested groups, or other objects.
vendor-user-memberships.csv A user-centered report showing where resolved vendor users are members.
vendor-user-accounts.csv An account inventory for resolved vendor users, including status and password-related AD fields.
vendor-group-discovery.html A browsable report grouped by domain, with confidence classes and reason tags. HTML output escapes directory metadata.
Console summary A quick terminal overview of counts by domain, confidence band, reason, and failed domains.

The report code includes two practical safety measures. Protect-CsvCell hardens CSV values against formula injection, and Write-HtmlReport HTML-encodes values because AD metadata such as names and descriptions can be influenced by people outside the script.

Quality checks

The tests describe the intended behavior

The tests/ folder is broad and practical. There are focused tests for string helpers, LDAP filter construction, input reading, match confidence, name/container/owner/member matching, result shaping, CSV and HTML output, and the top-level command. Each test file is named after the function it exercises, so the test for a function is easy to find. 

pwsh -NoProfile -Command "Invoke-Pester -Path ./tests -Output Detailed"

There is no live directory here, so two complementary mechanisms make the code testable on Linux. They exist only because the AD boundary is thin and everything downstream is pure.

Stubs plus mocking, for units tests/_TestHelpers.ps1 dot-sources all of src/ and defines no-op Get-ADGroup / Get-ADUser stubs only if they are absent. Unit tests then use Pester Mock to feed Get-AdDiscoveryData synthetic directory objects.
A simulated forest, for the whole pipeline tests/fixtures/ is a self-consistent four-domain forest — a directory.json plus input CSVs. Import-DiscoveryFixture.ps1 shapes it exactly like Get-AdDiscoveryData output, so the full engine runs end to end with no AD and no mocking.

The fixture doubles as the behavioral specification. tests/fixtures/README.md documents an exact expected result for every signal, and Fixture.Tests.ps1 asserts the engine against that oracle. If you change a scoring rule, add a report column, or adjust how candidates are fetched, read the matching *.Tests.ps1 and the fixture README first — together they usually explain the edge case more clearly than a comment in the code would.

Suggested route

A good way to read the codebase

If you are comfortable with basic PowerShell but not used to larger projects, do not start with the longest file. Read in this order so each layer has a purpose before you see the details.

README.md and Find-VendorAdGroup.ps1 Understand the command-line shape and the parameters users actually pass.
VendorAdGroupDiscovery.psm1 and src/Find-VendorAdGroup.ps1 See how the module is loaded and how the top-level function wires the work together.
src/Input/Read-DiscoveryInput.ps1 Learn the shape of the input object that every later layer receives.
src/Engine/Invoke-DiscoveryEngine.ps1 Follow the clean in-memory pipeline: find candidates, expand closure, filter, display-shape, sort.
src/Engine/Get-MatchConfidence.ps1 and match helpers Understand exactly what each kind of evidence is worth and how reasons are represented.
src/Ad/Get-AdDiscoveryData.ps1 Read this last. It is long because it balances correctness, performance, batching, caching, and AD query limits.
src/Report and related tests Finish with the output layer, where selected results are turned into useful files for review.

Mental model: the AD layer gathers a careful shortlist, the engine explains why each group belongs on that shortlist, and the report layer makes the explanation usable by humans.
